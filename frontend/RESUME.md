# 简历可写版本（Flow Canvas 前端）

## 项目：AI Flow Canvas（可视化工作流编排前端）

- **技术栈**：React 19、TypeScript（strict）、Vite、TailwindCSS、Zustand、@xyflow/react
- **项目简介**：在浏览器内提供可视化节点编排与 DAG 执行能力，支持全图运行、单节点运行，以及图片节点多图处理（AI 操作/生成视频/收藏/预览/加入会话）。

### 一句话简述（可直接放简历项目描述）

- 基于 React + ReactFlow 构建可视化工作流画布，前端实现 DAG 执行编排与可观测日志，落地图片/视频生成类异步任务的统一状态管理与交互约束（多图选择、自动连线、终止节点规则）。

### 个人职责与亮点

- **画布交互与可用性**
  - 基于 `@xyflow/react` 搭建节点画布，支持拖拽创建节点、连线、缩放/适配视图、小地图导航、框选多选与整体拖动
  - 通过 **Space + 拖拽平移** 与禁用 Shift 多选（统一用框选），解决平移/框选手势冲突，降低误操作率并提升上手体验

- **DAG 执行引擎（前端侧编排）**
  - 实现 Kahn 拓扑排序，校验非法边与环（cycle），保证流程可执行
  - 支持 **全量执行** 与 **目标节点依赖子图执行**（单节点 Run），减少无关节点重跑，缩短等待时间

- **图片/视频节点异步编排与体验优化**
  - 设计图片节点多图模型（`images[] + activeIndex + selectedIndexes`），以“最多 8 张网格 + 复选框/全选”完成多图选择输入，避免误对全量图片执行操作
  - 抽象 `aiTask` 统一表达异步任务（idle/running/success/error、整体/分项进度、错误信息、时间戳），支持局部重绘/消除笔/超清/文字重绘/微调/生成视频等动作
  - 任务产物采用“**结果节点**”承接：图片操作生成新图片节点并自动连线（image→image），视频生成新视频节点并自动连线（image→video）
  - 通过 UI（移除 source handle）+ 数据层（onConnect 兜底）双层约束，强制 **video 只能作为终止节点**，保证工作流语义稳定

- **统一状态与可观测性**
  - 使用 Zustand 收敛 nodes/edges、执行状态、日志、收藏、会话、预览等跨区域状态，降低组件耦合
  - 统一日志链路（DAG/节点/图片操作），对日志结果做容量限制与字段清洗，便于演示与排障

- **工程化与可维护性**
  - 组件化拆分画布页：`CanvasPage` 只做装配，左右面板与画布容器独立为 `LeftPanel / RightPanelShell / FlowCanvas`
  - monorepo（pnpm workspace）统一脚本，根目录一键并行启动前后端，降低演示/联调成本

### 结果与收益（可选）

- 提供接近设计工具的画布体验（框选/多选/缩放/导航），并具备可执行工作流能力
- 单节点依赖执行显著减少全图重跑开销，提升交互流畅度与迭代效率

## 面试 Q&A（问题 → 解决方案 → 亮点/收益）

### Q1：为什么 DAG 执行不放后端，而是前端也做一套执行器？

- **问题**：画布编辑阶段需要快速反馈（连线是否合法、是否存在环、单节点依赖执行范围），且演示/调试要求高可观测性。
- **解决**：前端实现 `executeDag / executeDagToTarget`：
  - Kahn 拓扑排序 + cycle 检测，提前阻断不可执行流程
  - 支持“全图执行 / 目标节点依赖子图执行”，减少无关分支重跑
  - 执行过程写入统一 logs（dag/node scope）
- **收益**：交互更像“可执行的设计工具”，单节点运行显著降低等待与资源浪费。

### Q2：图片节点为什么要做 `selectedIndexes`，而不是默认对当前 active 图或全量操作？

- **问题**：多图场景下，用户常需要“选几张一起做同一种操作”；默认全量容易误操作且不可控。
- **解决**：用 `selectedIndexes` 显式表达输入集合：
  - UI：最多 8 张网格 + 复选框/全选
  - store：`runImageNodeAction` **严格以 selectedIndexes 对应的 URL** 作为 `sourceImages`；未选择则直接提示并返回
- **收益**：任务输入可控、语义清晰，避免误对全量图片执行导致成本与体验问题。

### Q3：如何把“局部重绘/抠图/超清/视频生成”等不同异步任务的状态统一起来？

- **问题**：动作多、耗时长、可能多图并发，容易出现“状态散落在多个字段、UI 难以一致展示、错误难以定位”。
- **解决**：抽象 `aiTask`：
  - 统一 status（idle/running/success/error）+ progress
  - 支持 `items[]` 表示每张图的子任务状态，并聚合整体进度
  - 任务开始/结束写日志，失败写 globalError + aiTask errorMessage
- **收益**：UI 展示一致、可观测性更强，扩展新动作只需扩展 action 分支与任务编排。

### Q4：为什么“视频节点只能作为终止节点”，怎么保证约束不被绕过？

- **问题**：视频通常是终态产物，继续向下游连接会让语义与后续执行复杂度上升。
- **解决**：双层约束：
  - UI 层：视频节点不提供 source handle
  - 数据层：`onConnect` 中检查 source type 为 video 时直接拒绝新增边
- **收益**：即使未来 UI 调整或程序化新增边，也不会破坏工作流语义。

### Q5：遇到过哪些典型交互 Bug？怎么修的？

- **操作栏被裁剪/不出现**
  - **原因**：节点根容器使用了 `overflow-hidden`，导致浮动工具栏超出区域被裁剪
  - **修复**：移除根容器裁剪，改用局部容器控制溢出；同时提高 Handle zIndex 与偏移，避免被圆角/边框遮挡
- **画布平移与框选手势冲突**
  - **原因**：默认交互在拖拽/框选/平移间容易误触
  - **修复**：采用 Space + 拖拽平移；禁用 Shift 多选，统一用框选提升一致性
